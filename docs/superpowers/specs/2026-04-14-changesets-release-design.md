# Changesets Release Design

## Goal

Adopt the mainstream Changesets release workflow for the `ki` npm package so versioning, changelog generation, and npm publishing are driven by release PRs instead of manual tag-first publishing.

## Scope

This design covers:

- introducing Changesets configuration and scripts
- switching the repository from tag-first npm publishing to release-PR-driven publishing
- generating `CHANGELOG.md` from changesets
- updating GitHub Actions to use Changesets for release management
- updating contributor documentation for the new release flow

This design does not cover:

- monorepo multi-package release policies
- prerelease channels
- canary releases
- non-npm package registries

## Current State

- the project is a single npm package named `ki`
- CI currently publishes to npm from a pushed `v*` tag
- version/tag matching is enforced manually through a custom check script
- changelog generation is not automated

## Problems

1. The current tag-first workflow is not the mainstream Changesets model.
2. Changelog generation is manual or absent.
3. Versioning and publishing are coupled to human tag management.
4. The project wants a release flow closer to common open source Node package practices.

## Chosen Approach

Adopt the standard Changesets release PR workflow:

- contributors create changeset files for user-facing changes
- GitHub Actions maintains or updates a release PR on the default branch
- the release PR updates package version and `CHANGELOG.md`
- merging the release PR publishes the package to npm
- tags/releases are created from the published version flow rather than being the initial trigger

## Workflow Model

Day-to-day development:

1. make code changes
2. run `npm run changeset`
3. select package `ki`
4. choose release bump type
5. write a user-facing summary
6. commit code plus the generated changeset file

Release management:

1. changesets action runs on the default branch
2. if unreleased changesets exist, it opens or updates a release PR
3. the release PR contains:
   - version bump in `package.json`
   - updated `CHANGELOG.md`
   - consumed changeset files removed
4. merging the release PR triggers npm publish

## Repository Changes

The repository will add:

- `.changeset/config.json`
- `.changeset/README.md`
- `CHANGELOG.md`

`package.json` will add scripts:

- `changeset`
- `version-packages`
- `release`

## GitHub Actions Design

The existing tag-driven publish workflow will be replaced by a Changesets workflow that:

- runs on pushes to the default branch
- installs dependencies
- runs repository checks:
  - `npm run check`
  - `npm test`
  - `npm run build`
- invokes the Changesets GitHub Action

The Changesets action should:

- open or update a release PR when pending changesets exist
- publish to npm when the release PR is merged

Required secrets:

- `NPM_TOKEN`

GitHub permissions should allow:

- writing pull requests
- writing package/release metadata as needed by the action
- committing version/changelog updates back to the repository

## Versioning and Tagging

Manual `check:tag` enforcement will be removed.

Versioning source of truth becomes the release PR generated by Changesets.

Tags should be created from the published version flow, not by humans as the first step. The important contract becomes:

- merged release PR => version bump committed
- publish step uses that committed version

## Changelog Design

`CHANGELOG.md` will be generated and maintained by Changesets.

Each changeset summary should be written as end-user-facing release notes. Those entries are then aggregated into the versioned changelog.

This provides:

- reproducible release notes
- a clear contributor workflow
- changelog entries tied directly to shipped changes

## Contributor Workflow

Contributors should do the following for user-visible changes:

```bash
npm run changeset
```

Then commit the generated `.changeset/*.md` file with the code change.

For internal-only changes, the team can decide whether to skip a changeset. Preferred starting rule:

- require changesets for anything user-visible
- allow skipping for docs-only or CI-only maintenance changes

## Documentation Design

README should explain:

- installation remains `npm install -g ki-skill`
- development commands remain npm-based
- releases are managed through Changesets release PRs

Contributor-facing docs should explain:

- when to create a changeset
- how to write a good changeset summary

## Error Handling

The release workflow should fail if:

- checks fail
- changesets action cannot open/update the release PR
- npm publish fails

The repository should not publish directly from arbitrary tags anymore.

## Testing Strategy

Implementation should verify:

- Changesets config exists and is valid
- `npm run changeset -- --help` works
- CI workflow contains Changesets action and no longer depends on tag-first publishing
- package scripts reference Changesets commands
- `CHANGELOG.md` is present and tracked

Because GitHub-hosted release PR creation and real npm publish cannot be exercised locally here, local verification will stop at config, scripts, and workflow contract checks.

## Risks

- Team members may forget to add changeset files at first.
- Replacing tag-first publishing changes established release habits.
- Poorly written changeset summaries will degrade changelog quality.

## Mitigations

- document the workflow in README
- keep scripts obvious and simple
- rely on the release PR as a review checkpoint for version and changelog quality

## Implementation Outline

1. add Changesets dependency, config, scripts, and baseline changelog files
2. replace tag-first release workflow with Changesets GitHub Action workflow
3. remove now-obsolete tag-version enforcement script
4. update README to explain the Changesets release model
5. run local verification for scripts and workflow contracts
